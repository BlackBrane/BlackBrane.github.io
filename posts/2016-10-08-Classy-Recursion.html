A classy approach to domain specific languages

Posted on October 8, 2016 by Cliff

How best to work with domain specific languages has long been a matter or personal and professional interest to me. After doing a lot of learning, I’ve implemented a prototype program that applies some of the promising techniques I’ve picked up, with some variations that I haven’t yet seen elsewhere.

This project, available on Github, aims to make an extensible library and interpreter for manipulating mathematical expressions and proofs about them. I especially wanted to make it easy to extend with new mathematical constructs when the need arises. Therefore, we need much of our logic at an abstract level, which in Haskell land means type classes.

Review of catamorphisms in Haskell

To start off, I’ll briefly review some basic ideas about recursion schemes first described in Bananas, Lenses, Envelopes, and Barbed Wire. For now I will only be focusing on catamorphisms. If you aren’t familiar with them I recommend checking out the blog series An Introduction to Recursion Schemes.

For the sake of discussion lets consider this very simple data type.

data Expr = Lit Int
          | Plus Expr Expr
          | Mult Expr Expr

The data type could be anything that has a recursive definition. The idea is that we have expressions or arbitrary size and we’re going to show a nice way to perform recursive computations on them.

An example of the straightforward way to do this would be as follows.

eval :: Expr -> Int
eval (Lit n)    = n
eval (Plus x y) = eval x + eval y
eval (Mult x y) = eval x * eval y

There is nothing wrong with this basic approach, particularly when the example is so simple, but when the complexity begins to grow it can be extremely helpful to use a recursion scheme that takes care of all the recursive function calls.

First we rewrite our data type in a polymorphic form, making every constructor that referred to itself instead refer to a type variable.

{-# LANGUAGE DeriveFunctor #-}

data Expr a = Lit Int
            | Plus a a
            | Mult a a
            deriving Functor

The motivation for this is not to have to describe our computation all at once, but instead to be able to describe it as a single recursive step, which is then applied repeatedly. The polymorphic type variable in this definition can represent the return type of any particular function we may wish to create.

First, though, how do we represent an arbitrarily recursive expression with this type? Seemingly this would entail the infinite type

Expr (Expr (Expr ... ))

This obviously is not valid Haskell. But we can accomplish the same thing with the following data type.

newtype Fix f = In { out :: f (Fix f) }

It may take some practice to get used to but now we can represent our arbitrarily nested expressions just as before.

ex1 :: Fix Expr
ex1 = In (Plus (In (Lit 2)) (In (Lit 2))) -- 2 + 2

Now we’re ready to re-cast our computation. As stated, all we really need to do is define the computation at a particular nesting layer. Intuitively, we can also think of it as applying to the last layer.

evalStep :: Expr Int -> Int
evalStep (Lit n)    = n
evalStep (Plus x y) = x + y
evalStep (Mult x y) = x * y

You can see that this definition is almost the same as the basic eval function defined above, except that rather than calling itself recursively in the function body, it instead uses the already-evaluated values brought into scope by the pattern match.

Since every recursive function we define will be built in the same way from these “step” functions, we might as well give them a name.

type Algebra f a = f a -> a

Now the magic of this catamorphism scheme allows us to turn any such computation step into the full thing.

cata :: Functor f => Algebra f a -> Fix f -> a
cata g = g . fmap (cata g) . out

Our full evaluation function is now simply:

eval :: Fix Expr -> Int
eval = cata evalStep

There is also a dual notion of anamorphsim, genearlizing unfolds whereas catamorphisms are a generalization of folds.

type Coalgebra f a = a -> f a

ana :: Functor f => Coalgebra f a -> a -> Fix f
ana coalg = In . fmap (ana coalg) . c

Full Flexibility

In writing my library I found that, while sometimes the above approach was amazingly helpful, other times I just wanted to work with a vanilla data type. After all, if we aren’t doing this fancy recursion scheme stuff it can be a pain to work with these extra In constructors all over the place.

Well one straightforward way to have it both ways is to define both versions of our data type and relate them via a type class.

Lets adopt the following definitions:

data Expr = Lit Int
          | Plus Expr Expr
          | Mult Expr Expr

data Expr_ a = Lit_ Int
             | Plus_ a a
             | Mult_ a a
             deriving Functor

Eventually, it may be nice to allow the user to specify their language just in the first non-parametric form and then have the second form generated by Template Haskell.

In any event, once we have these definitions we can relate them with a type class, essentially making an isomorphism with a little help from the recursion infrastructure we introduced earlier.

{-# LANGUAGE MultiParamTypeClasses,
             FunctionalDependencies #-}

class Functor f => Fixable f a | a -> f, f -> a where
  foldFix :: Algebra f a
  unfoldFix :: Coalgebra f a

  fromFix :: Fix f -> a
  fromFix = cata foldFix

  toFix :: a -> Fix f
  toFix = ana unfoldFix

We can instantiate our example as:

instance Fixable Expr_ Expr where
  foldFix (Lit_ x)      = Lit x
  foldFix (Plus_ x1 x2) = Plus x1 x2
  foldFix (Mult_ x1 x2) = Mult x1 x2

  unfoldFix (Lit x)      = Lit_ x
  unfoldFix (Plus x1 x2) = Plus_ x1 x2
  unfoldFix (Mult x1 x2) = Mult_ x1 x2

And now we can operate even more painlessly

catamap :: Fixable f a => Algebra f b -> a -> b
catamap alg = cata alg . toFix
